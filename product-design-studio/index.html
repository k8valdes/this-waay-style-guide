01 · Architecture

Token architecture

One flat layer, on purpose. Every brand value is a single --tw-* custom property, declared once in this page's stylesheet and mirrored one-to-one in tokens.json — color (with its accessible ink text variants), gradients, type family, weight, the 10-step size scale, tracking, leading, the 8px spacing rhythm, radii, shadows, and section padding. Components reference tokens directly; there are no semantic or component alias layers.

Flat by design

Value → use, no alias hop. Representative declarations from this page's :root:
--tw-navy: #0C2A48;              /* color primitive — used directly */
--tw-green-ink: #2C6D4D;         /* accessible text variant of green — 6.2:1 on white */
--tw-grad-soft: linear-gradient(135deg, #EEF8EC 0%, #F2FBFB 100%);  /* standard light surface */
--tw-text-md: 0.85rem;           /* step 4 of the 10-step UI text scale */
--tw-radius-card: 14px;          /* card and panel corners */
--tw-section-pad: clamp(56px, 8vw, 110px);  /* section rhythm */

This flatness is a deliberate per-brand difference: the Gridmark Partners sibling system documents an explicit three-layer model (primitive → semantic → component), while This Waay's block stays flat — each reflects that document's own authoring history. See RELATIONSHIP.md in the unified home before "aligning" one to the other.

05 · Foundations

Motion

Motion should feel like the flight path itself — purposeful and directional, never decorative-for-its-own-sake. Every animated component ships with its own reduced-motion fallback; this page's global reset (see below) is the safety net, not the only line of defense.

Duration & easing scale

Micro · 120–160msease-out — hover/focus feedback (buttons, swatches, links)
Element · 200–300msease-out — card lift, menu open/close, tab switch
Reveal · 400–600msease-out — one-directional scroll reveals, the flight-path draw-on below

Nothing runs slower than ~600ms without a specific reason. No bounce, spring-overshoot, or elastic easing — it reads as playful, not composed.

Signature moment — the flight-path draw-on

A one-time stroke-dashoffset draw, ~2.4s ease-out, for hero and section entrances — never looping. With prefers-reduced-motion: reduce, the path renders complete immediately (try it: enable reduced motion in your OS and reload).

May animate
  • Hover/focus micro-feedback — the existing swatch lift and button brightness are the reference examples
  • One-directional scroll reveals: fade + a small (≤16px) translate, once, never re-triggering on scroll-up
  • The flight-path draw-on for hero/section entrances
  • Loading indicators, with a text alternative for screen readers (not motion alone)
Must never
  • Parallax layers moving at conflicting speeds
  • Autoplaying or looping background motion with no pause control (2.2.2)
  • Anything flashing more than 3 times per second (2.3.1)
  • Motion as the only way information is conveyed (e.g., a shake to signal "error" with no text)

Reduced motion — the contract

This page's global reset removes animation/transition duration for prefers-reduced-motion: reduce. Every new animated component adds its own explicit fallback next to the animation itself (see .swatch:hover, .btn:hover, and .flight-draw above) — don't rely on the blanket reset alone, since a future removal of the global rule shouldn't silently break anyone's accessibility.

Slide-deck note: transitions between slides are a simple fade or cut only — no per-slide novelty transitions. Advance is always presenter-controlled; never auto-advance a deck. Full deck templates are in Slide decks.

06 · Foundations

Accessibility

WCAG 2.2 AA is the baseline for every This Waay and client deliverable — not a follow-up pass. It's applied here, in the guide itself, so every rule below is demonstrated live rather than just described. A few of the studio's own signature colors fail AA contrast when used as text on light grounds — the fixes are documented as tokens below and used consistently throughout this page.

Color & contrast

The saturated brand hues (Waay Green, Teal, Slate) are fill and dark-ground colors — several drop below 3:1 when used as text on white or soft-gradient grounds. Use the paired "ink" token for text in those cases; the fill stays exactly as documented in Color system.

Waay Green#54B9872.4:1 on whiteFails text — fill/icons/dark grounds only
Green — Ink--tw-green-ink6.2:1 on whitePasses AA — eyebrows, links, inline emphasis on light
Teal#0BABAB2.8:1 on whiteFails text — fill/dark grounds only
Teal — Ink--tw-teal-ink5.7:1 on whitePasses AA — subheads, links on light
Slate#8DA3B52.6:1 on whiteFails text — borders/gridlines/dark grounds only
Ink — Muted--tw-ink-muted5.1:1 on whitePasses AA — captions, secondary labels on light
No — Ink--tw-no-ink6.6:1 on whitePasses AA — "don't" tags, form error text
Navy / White#0C2A48 / #FFFFFF16.6:1Passes AAA — always safe either direction

WCAG 1.4.3 / 1.4.11 — 4.5:1 minimum for normal text, 3:1 for large text (≥24px, or ≥18.66px bold) and meaningful graphical objects/UI boundaries.

Focus states

Then me Press Tab to move focus — every interactive element on this page shows a visible ring (:focus-visible, 3px, offset 3px). Nothing relies on the browser default.

WCAG 2.4.7 / 2.4.11 — one shared focus recipe, swapped to Green Bright on navy/hero grounds where Green Ink wouldn't show.

Structure & keyboard

Required on every page
  • A skip-to-content link as the first focusable element (this page has one — tab from the very top)
  • One <main> landmark, one <h1>, sequential <h2>s — never skip a level
  • Every clickable thing is a real <button>/<a>, or has role + tabindex="0" + a keydown handler for Enter/Space
  • Async confirmations (copy, save, submit) update an aria-live="polite" region — a visual-only state change (like the old "Copied" tag) is invisible to screen readers
  • Purely decorative SVGs/icons get aria-hidden="true"; meaningful ones get role="img" + aria-label
Never
  • A clickable <div> with only an onclick — not reachable or operable by keyboard
  • Color as the only signal (link-only-by-color text, color-only chart series, color-only form errors)
  • Content or navigation that requires a mouse (drag-only reordering, hover-only menus with no focus equivalent)
  • Auto-advancing carousels, slides, or background motion without a visible pause control

Images, icons & illustration

The paper-cut landscape illustrations (see Illustration) are narrative, not decorative, when used as a hero or section image — they need real alt text describing the metaphor, not the literal scene.

Good alt text
  • "Illustration of a suspension bridge crossing a canyon at night, representing the studio's approach to system integration"
Not this
  • "Bridge, mountains, stars, moon, coral dots" (literal, no meaning) — or a blank alt="" on a meaningful hero image

Forms & data

Form patterns (labels, required-field convention, inline error state) are documented in Components — every field has a visible, programmatically-associated <label>, and errors are announced via aria-live, not color alone. Charts and diagrams never rely on color alone either — see the secondary-encoding example in Data visualization.

Motifs & iconography

Three recurring devices tie everything together: the paper plane, the dotted flight path, and angled section breaks.

The paper plane (official geometry)

Exact path data from the master logo file. The wing is the official gradient #62D38C → #30737B (light at the nose, deep teal at the tail, running diagonally). Body is navy on light grounds, white on navy — never outlined, never flat green. Faded oversized silhouettes (6–10% opacity) texture backgrounds in both modes.

Dotted flight path

A — Single loop

B — S-wave

C — Tight coil

D — Wide arc

E — Double curl

On light ground — navy stroke

Trails are round dots, not dashes (SVG: stroke-dasharray: 0.1 8, round caps, ≈3px weight), and every trail includes at least one loop-de-loop curl — hand-flown, never straight. White on navy grounds; navy on light grounds. The trail exits the plane's tail and extends behind it. Use A–E as canonical references; new trail shapes should follow the same character.

Duotone icon set

Bold navy outlines (≈3.5px at 48px grid) with a single flat green fill element. On navy backgrounds, outlines invert to white; green fills stay green.

Angled section breaks

Sections meet on a shallow diagonal (2–4°), alternating navy and soft-gradient surfaces. Echoes the plane's wing line.

10 · Application

Screenshots & product imagery

The most-used visual in the studio's work — and the backbone of the AI Guidance case studies, the studio's key marketing asset. Product screenshots are never dropped in raw; they're always framed, grounded, and annotated in one of two canonical treatments.

Treatment A · Light ground

Default for in-flow case-study and work-tile screenshots. Soft-gradient or white ground, 8px screenshot corners, soft navy shadow (rgba(12,42,72,0.18)). Real product aspect ratio — never stretched.

Treatment B · Navy hero + callouts

AI panel

For case-study heros and feature spotlights. Navy ground with faint topographic contour lines, screenshot floated with shadow, green callout pills and boxes marking the relevant UI region. Callouts are always Waay green — the only annotation color.

Do
  • Frame every screenshot — light gradient ground or navy hero
  • Consistent 8px corner radius and one soft navy shadow
  • Annotate with green pills, boxes, and thin connector lines only
  • Use phone/browser mockup frames for mobile and responsive work
  • Preserve true product aspect ratio
Don't
  • Drop in raw, unframed, full-bleed screenshots
  • Use heavy black drop shadows or mismatched corner radii
  • Annotate in red, orange, or any non-brand color
  • Stretch, skew, or crop away meaningful product context
  • Stack more than 2–3 callouts on one screenshot

Diagrams & frameworks

Slide decks

Six templates cover most decks: title, section divider, content, quote, chart, and closing. Gray deck headers, teal-ink subheads, green category labels, and Axiforma throughout — the same system as the rest of the brand, adapted for a projected 16:9 surface.

Deck rules

Aspect ratio16:9Safe area: keep text ≥5% inset from every edge for projector/crop tolerance
Slide numberingbottom-rightOmit on title and closing slides
Presenter notesbelow the slide, never on itSpeaker context and talk track live in the notes field, not on-slide
Minimum text size~18px equivalent at 1080pProjected/shared-screen contexts need larger minimums than web body text
Contrastsame as webWCAG 2.2 AA still applies — see Accessibility
Transitionsfade or cut onlyNo per-slide novelty transitions; advance is always presenter-controlled — see Motion

Any chart or diagram embedded in a deck needs the same alt text/description discipline as on the web — if the deck is shared as a PDF or read aloud, the data still needs to be conveyed in words somewhere (a takeaway line, a table, presenter notes).

How AI tools should use this guide

Instructions for AI assistants generating This Waay assets:
  1. Use the CSS custom properties in this file's :root block as canonical tokens — never approximate brand colors from memory.
  2. One green: #54B987 is the only green for buttons, labels, icons, and recurring UI on any ground. Punch Green (#3DD68C) may appear at most once per asset, as a deliberate emphasis moment — if in doubt, don't use it. (This page uses its one punch on the hero headline.)
  3. For any illustration or image prompt, start from the base style block in section 04 and modify only the scene variables. Never generate the deprecated isometric style.
  4. Headlines in sentence case; at most one green/teal highlight per headline; eyebrows are the only uppercase element.
  5. Typography is Axiforma in all assets; substitute Poppins only when Axiforma is unavailable. Never introduce a second family.
  6. Apply the voice rules in section 06 to all copy. When in doubt: shorter, calmer, more structural.
  7. Slides follow the deck system: gray headers (#595959), teal subheads, green category labels, white or navy grounds — set in Axiforma going forward (legacy deck face is deprecated).
  8. Paper plane motif: use the exact SVG paths embedded in section 05; wing gradient is #62D38C → #30737B (nose → tail), body navy on light / white on navy. Flight trails are round white or navy dots with at least one loop — never straight, never dashed.
  9. Charts use the fixed series order (green, steel, teal, navy, slate); single-series is always Waay green; no chart-junk, 3D, or non-brand colors.
  10. Screenshots are always framed — light gradient ground or navy hero with topographic lines — with 8px corners and a soft navy shadow; annotate only in Waay green.
  11. Spacing follows the 8px scale; content maxes at 1080px; sections alternate navy/soft on shallow angled breaks with a centered eyebrow-H2-lede header.
  12. Diagrams use bold navy outlines, one green accent each, and the fixed double-diamond color order (slate, navy, steel, green).
  13. Coral (#F2A08D) appears only inside illustrations — never in UI, charts, or text.
